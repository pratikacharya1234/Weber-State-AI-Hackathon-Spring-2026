Why we built it

Students ask the same
questions every semester.

"Where do I register?"  "What's the prereq for CS 2420?"  "Where is Tracy Hall?"
These questions get asked at 11pm when nobody is online. The WSU website has the answers but it takes 10 minutes to find them.

The existing chatbots either don't know WSU exists or hand you a 404 link. We wanted something that actually knows Weber State — not a wrapper around someone else's model, not a keyword search. A real language model trained on real WSU data that can answer in plain English.

Neither of us had trained a 600M+ parameter model before. We decided to build one anyway. This is what happened.

Before the hackathon started

This was attempt five.

Attempt 1 — Rust model from scratch
Built a transformer in pure Rust before using any ML framework. Generation produced grammatically structured noise. Discarded.
Attempt 2 — Rust with ferrum (FERRUM_REPO)
Built a custom autograd engine (Ferrum) in Rust. Training worked but was too slow and the generated text was weak. Scrapped the training side, kept it for inference.
Attempts 3 and 4 — Variations
Tried different architectures and smaller models. Generation quality never got past incoherent output. All four ended the same way: started over.
Hackathon — Attempt 5
628M parameter transformer in Python/PyTorch. Rust kept for inference only. This is the one that actually worked.
During the hackathon itself
We also restarted twice during the hackathon.

First restart: tokenizer mismatch — the model had learned a different vocab than what the inference code expected.

Second restart: training diverged after changing the learning rate schedule mid-run. Went back to the last good checkpoint.
What we kept from the failures
The Rust inference engine (Titanium) from attempts 2-4.
The 16,384-token BPE tokenizer trained on WSU text.
The understanding that generation quality comes from data, not architecture tricks.
Dipendra's work

The dataset.

62,042 WSU Q&A pairs. Scraped from the WSU course catalog, student handbook, department pages, housing portal, financial aid documentation, and campus maps. Deduplicated, cleaned, and formatted as Q:/A: pairs to match the training format exactly.

The format is the most important part. During training we found that the model learns the question-to-answer relationship just as much as the answer content. Mismatching the prompt format at inference caused the most hallucinations we saw.

The TF-IDF index runs on all 62K pairs at query time. When someone asks a question, the closest Q&A pair in the dataset is retrieved and injected as a one-shot example in the same Q:/A: format the model was trained on.

The model design.

Decoder-only transformer. Weights initialized to random noise on day one. Every parameter value in this model came from gradient descent on our training data. No pretrained weights, no transfer learning.

Structure

Type: GQA

KV heads: 8
KV cache: 3x smaller

File: wildcat_16k.json

GQA uses 8 KV heads instead of 24, cutting KV-cache memory by 3x during generation. This was inspired by LLaMA 2. Critical for running on the VM while other processes are active.

Weight tying means the embedding matrix and the final output projection share the same weights. This saves ~75M parameters and improves training stability, but caused a crash when saving GRPO checkpoints — PyTorch detected the shared memory and threw an error. Fixed by cloning all tensors before saving.

# Why weight tying caused the GRPO save crash # tok_emb.weight and head.weight point to # the same memory. safetensors rejects this. # Fix: clone everything before saving state = {k: v.clone() for k, v in model.state_dict().items()} save_file(state, 'weights.safetensors')

Where training started

Phase 1: 1.1 billion tokens
on Google Colab.

The first phase of training ran entirely on Google Colab with an A100 GPU. All data was stored on Google Drive — Colab can't persist large datasets between sessions, so the Drive was the only way to keep training state.

The Phase 1 data was pre-tokenized into binary shards using the 16K BPE tokenizer and stored as wildcat_train_*.bin files in the Drive folder WildcatAI_Final/data/08_shards_16k. Each shard is ~100MB.

The 1.1B tokens gave the model basic English fluency — correct grammar, punctuation, word associations. Without this foundation, the WSU-specific fine-tuning phases would have had nothing to build on.

Phase 7 — what we tried

GRPO: reward-based
fine-tuning.

After Phase 6 we tried Group Relative Policy Optimization — the same technique used by DeepSeek-R1 and Kimi K1.5. The idea: generate multiple answers to the same question, score each one, and train toward the higher-scoring ones.

Our reward function scored on WSU-specific accuracy, not general quality. Mentioning real WSU details got a +0.5 bonus. Hallucinating non-existent facts got a -1.0 penalty. Best reward achieved: +1.238.

The problem: with only 200 training steps, GRPO didn't have enough data to generalize. The model got better at the training questions but started giving worse answers on anything slightly different. We reverted to the Phase 6 checkpoint for production.

# GRPO reward function G = 4 # completions per question MAX_NEW = 40 # tokens per completion # Score each completion def reward(text): score = 0.0 for fact in GOOD_WSU_FACTS: if fact in text.lower(): score += 0.3 # max +1.0 for pattern in BAD_PATTERNS: if pattern.search(text): score -= 1.0 # hallucination penalty if len(text.split()) < 10: score -= 0.5 # too short return score # Best training reward: +1.238 # TOTAL_STEPS: 200 (not enough) # Result: degraded on out-of-domain

The WildcatAI orchestrator runs in Rust using Axum. It receives the query, calls the TF-IDF RAG server to retrieve the best matching Q&A pair, builds the Q:/A: prompt, sends it to Titanium for generation, then cleans the response.

Titanium is the inference engine — pure Rust using HuggingFace's candle library. It loads the safetensors weights, runs the forward pass in bfloat16 on the A100, and streams tokens back. No Python at inference time.

What we disabled
AXIOM (HDC hyperdimensional computing) — built and tested, but bag-of-words vectors returned wrong matches. Searching "register" returned Criminal Justice courses.

AEGIS (cognitive engine) — built, replaced entirely by TF-IDF RAG which gave better recall accuracy.

The fix for generation quality

We built a guard layer
on top of the model.

Exact-match knowledge base
25 hardcoded answers: building locations, phone numbers, addresses, course descriptions. These never go to the model. Zero latency, never wrong.
Filler text removal
The model often starts with "That's a great question!" or "Well, basically..." — phrases that add nothing. Stripped before showing to the user.
Repetition detection
Two checks: same 15-word phrase appearing twice in the same response, and unique-word ratio below 30%. Both truncate at the first repeat.
Off-topic detection
Spaced-letter patterns like "a b c d e f". References to Yale or Harvard in a WSU answer. Both return a safe response with real contact information instead.
// Unique-word ratio — catches // "Registrar... Registrar... Registrar..." style loops function isHighRepetition(text) { const words = text.split(/\s+/) const n = words.length if (n < 20) return false const unique = new Set( words.map(w => w.toLowerCase() .replace(/[.,!?;:]$/, '')) ) const ratio = unique.size / n if (n < 50) return ratio < 0.4 return ratio < 0.3 } // Stop sequences that cut generation early const STOP = [ '\n\n\n', 'John F. Moench became', ]

Honest reflection

What we'd change
if we did it again.

Larger context window
512 tokens is too small. Multi-turn conversations break immediately because previous messages get cut off. Need at least 2048 to be genuinely useful.
Deploy the prompt format fix
The Q:/A: fix is compiled and ready. The binary is sitting at target/release/wildcatai. One scp command once SSH is unblocked. This would fix most remaining hallucinations.
Streaming inference
Users wait 5-8 seconds with no feedback. Token-by-token streaming would make it feel instant. Requires server-sent events in the Rust server and a reader in the frontend.
More GRPO training steps
200 steps wasn't enough. With 2,000+ steps and a better reward function, GRPO could have improved accuracy without degrading. We ran out of time.

The model itself works. It answers WSU questions correctly when the RAG retrieves the right context and the prompt is formatted correctly. The remaining issues are all infrastructure — the wrong prompt format on the VM, no streaming, SSH being blocked.

The biggest thing we learned: training 628M parameters is the easy part. Getting the full system to work reliably — correct prompt format, stable serving, no hallucination loops — is harder than the training itself.

We built a quality layer in the frontend to handle what the model can't do yet. That tradeoff is what production ML actually looks like.
